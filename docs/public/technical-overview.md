# Technical Overview

This note documents the main technological and implementative choices behind the public Antica Fiamma shell.

It is written for a Digital Humanities reader who needs to understand how the project is structured, not for a full software audit.

## Architecture In Brief

Antica Fiamma is currently organized as a static front-end shell with separately generated runtime data.

The public repository contains:

- static HTML pages
- CSS and JavaScript assets
- public research-room assets
- generated static authority pages
- selected build scripts
- deployment packaging
- smoke checks
- selected source-capture utilities

The heavy runtime data payloads used by the live interface are maintained outside the public repository.

## Front-End Shell

The main public shell lives under:

```text
demo/frontend/
```

The current browser entrypoint is:

```text
demo/frontend/static/app.js
```

The interface also uses a split module layer under:

```text
demo/frontend/static/modules/
```

Important module areas include:

- `core/`: configuration, routing, shared state, loaders, DOM helpers, and runtime contracts.
- `panels/records_panel.global.js`: commentary-card and record-reading surface.
- `panels/authority_panel.global.js`: authority-layer interactions.
- `panels/coverage_panel.global.js`: canto and line entry.
- `panels/word/word_level_panel.global.js`: Dante Word Locus Layer and word-centered panels.
- `panels/line/line_level_panel.global.js`: line-level Cross-Canto Echoes and related line panels.

This split is still pragmatic rather than final. It reflects stabilized reading surfaces while preserving the existing public shell.

## Runtime Data

The public shell expects runtime data to be available through a data base path. In local development this can be `/data`; in production the current configuration can point to an external object-storage layer.

The public repository excludes:

```text
demo/frontend/data/
demo/frontend/reports/
demo/frontend/data_snapshots/
demo/frontend/data_legacy_pre_page_state_v2/
```

This keeps GitHub from functioning as a data dump and separates the public interface shell from the heavier generated corpus.

## Build Logic

The main data-build entrypoint retained for transparency is:

```text
demo/build_demo_data.py
```

This script prepares front-end-facing structures from local source data and derived layers. It includes logic for line payloads, commentary records, research layers, authority data, Dante word loci, and related outputs.

The generated runtime outputs are not versioned in the public repository.

## Authority Static Pages

Static authority pages are generated by:

```text
demo/build_authority_static_pages.py
```

The generated public pages live under:

```text
demo/frontend/autore/
demo/frontend/personaggio/
```

These pages are part of the public interface because they make author, work, and personaggio paths linkable outside a live reading session.

## Deployment

The public shell is deployed through Cloudflare Pages.

The packaging script is:

```text
deployment_output/PREPARE_PAGES_SHELL.py
```

It prepares:

```text
deployment_output/pages_shell_build/
```

That generated folder is ignored by Git. The GitHub Actions workflow runs the packaging script and deploys the resulting shell.

Workflow:

```text
.github/workflows/deploy-pages-shell.yml
```

## Local Preview

The local preview server is:

```text
demo/server.py
```

From the repository root:

```bash
python3 demo/server.py
```

Then open:

```text
http://127.0.0.1:8777/
```

The local preview may not reproduce the full live site unless the external runtime data layer is available.

## AI-Assisted Development

The project has been developed autonomously with AI-assisted coding support.

For DH documentation purposes, the important point is not the label "vibe coding" but the workflow discipline now being added around the prototype:

- source control through Git
- public repository boundary
- explicit data exclusion rules
- smoke checks for selected interface paths
- manual review of generated or AI-assisted changes
- progressive documentation of interface, data, rights, and deployment choices

The present documentation work is part of that consolidation. The goal is to make the project inspectable by others rather than dependent on the author's memory of how the prototype was assembled.

AI-assisted coding has supported implementation, refactoring, testing, and documentation work. It does not replace scholarly judgment, source attribution, rights decisions, philological framing, or final review of public-facing claims.

## Current Technical Scope

The current public repository is scoped as an inspectable public shell rather than a full corpus distribution.

The scope is defined by several current choices:

- heavy runtime data is external to the repository
- source and permissions status is documented in the public notes
- generated deployment output is ignored rather than versioned
- selected generated static pages are included when they are part of the public interface
- internal research workspaces remain outside the public repository
- public documentation is updated as domains, research rooms, and interface layers stabilize

This scope makes the repository readable as a DH project shell. It shows the interface, deployment path, selected build logic, and maintenance boundary while keeping generated payload distribution separate.
